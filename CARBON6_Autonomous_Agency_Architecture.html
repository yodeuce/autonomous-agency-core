I. Executive Overview

Document Purpose

This Technical Architecture Specification provides comprehensive documentation for the CARBON[6] Autonomous AI Agency framework—a production-grade infrastructure for deploying safe, aligned, and effective autonomous AI agents in enterprise environments.

The framework implements rigorous decision-theoretic principles through 24 interconnected modules spanning decision-making, memory management, environment modeling, value optimization, learning, and governance.

1.1 Purpose and Scope

The CARBON[6] Autonomous Agency framework addresses the fundamental challenge of deploying AI agents that can:

  • Operate Autonomously — Make decisions and take actions without constant human oversight
  • Maintain Alignment — Remain aligned with human values and organizational objectives
  • Ensure Safety — Operate within defined boundaries with deterministic constraint enforcement
  • Optimize Value — Maximize expected monetary value while managing risk appropriately
  • Learn Continuously — Improve performance through experience while preserving alignment
  • Provide Transparency — Maintain complete audit trails for all decisions and actions

1.2 Architectural Philosophy

The architecture is grounded in three foundational principles:

Markov Decision Process (MDP) Foundation

All agent behavior is modeled as a formal MDP with clearly defined state spaces, action spaces, transition dynamics, and reward functions. This provides mathematical rigor and enables formal verification of agent properties.

Expected Monetary Value (EMV) Optimization

Decisions are made by computing expected monetary values across all possible outcomes, weighted by probability and transformed through appropriate utility functions. This ensures economically rational decision-making.

Hierarchical Constraint Architecture

A multi-layered constraint system enforces safety at every level—from hard constitutional constraints that can never be violated, to soft operational guidelines that can be overridden with appropriate authorization.

II. Core Decision Subsystem

The Core Decision Subsystem provides the mathematical foundation for all agent decision-making. It implements a complete Markov Decision Process specification with policy definitions and reward modeling.

2.1 MDP Specification

core_decision/mdp_spec.yaml

The MDP specification defines the formal structure of the agent's decision problem. It serves as the "constitutional document" for agent behavior.

State Space Definition

The state space is partitioned into observable and latent components:

Observable States Description Type
task_status Current task execution state Categorical
resource_availability Available computational and external resources Continuous [0,1]
constraint_status Current constraint satisfaction state Boolean
time_remaining Time until deadline or timeout Continuous
Latent States Description Inference Method
user_intent Inferred user objectives Bayesian inference
environment_stability Estimated environment volatility Kalman filtering
task_complexity Estimated difficulty of current task Feature-based estimation

Action Space

The action space defines all possible agent actions, organized hierarchically:

OBSERVE Gather information from environment without modifying state
ANALYZE Process and analyze available information
RECOMMEND Generate recommendation for human approval
EXECUTE Take autonomous action within authorized scope
ESCALATE Escalate decision to higher authority
DEFER Postpone action pending additional information

Transition Model

The transition model specifies how actions affect state:

P(s' | s, a) = Σ P(s' | s, a, θ) * P(θ | history)

Where:
  s'    = next state
  s     = current state
  a     = action taken
  θ     = environment parameters (learned from history)

Discount Factor

The discount factor γ = 0.95 balances immediate and future rewards, with dynamic adjustment based on urgency:

γ_effective = γ_base * (1 - urgency_factor)

Where urgency_factor ∈ [0, 0.3] based on deadline proximity

2.2 Policy Definition

core_decision/policy_definition.py

The policy module defines how agents select actions given states. Multiple policy types are supported:

Policy Types

DeterministicPolicy

Maps states directly to actions via a lookup table or function. Used when optimal action is unambiguous.

π(s) = a*  where a* = argmax_a Q(s, a)
StochasticPolicy

Maps states to probability distributions over actions. Enables exploration and handles uncertainty.

π(a|s) = exp(Q(s,a)/τ) / Σ_a' exp(Q(s,a')/τ)

Where τ = temperature parameter controlling exploration
EpsilonGreedyPolicy

Balances exploitation and exploration with configurable epsilon decay.

π(a|s) = {
    1-ε+ε/|A|  if a = argmax Q(s,a)
    ε/|A|      otherwise
}

ε_t = max(ε_min, ε_0 * decay^t)

Policy Manager

The PolicyManager orchestrates policy selection and enforcement:

class PolicyManager:
    def select_action(self, state: State) -> Action:
        # 1. Pre-filter actions through constraint engine
        valid_actions = self.constraint_engine.filter(state, self.all_actions)

        # 2. Apply active policy to select from valid actions
        action = self.active_policy.select(state, valid_actions)

        # 3. Log decision for audit
        self.decision_log.record(state, action, self.active_policy)

        return action

2.3 Reward Model

core_decision/reward_model.py

The reward model defines the objective function that guides agent behavior.

EMV Reward Model

The primary reward model computes Expected Monetary Value:

R(s, a, s') = Σ_o P(o | s, a, s') * V(o)

Where:
  o = outcome
  P(o | s, a, s') = probability of outcome given transition
  V(o) = monetary value of outcome

Risk-Adjusted Reward Model

Incorporates risk preferences through utility transformation:

R_adjusted(s, a) = U(EMV(s, a)) - λ * Risk(s, a)

Where:
  U() = utility function (e.g., CRRA)
  λ = risk aversion coefficient
  Risk() = risk measure (VaR, CVaR, variance)

Multi-Objective Reward

Supports multiple objectives with configurable weights:

R_total = Σ_i w_i * R_i(s, a)

Subject to: Σ w_i = 1, w_i ≥ 0
Objective Weight Range Description
Task Completion 0.3 - 0.5 Primary task success
Resource Efficiency 0.1 - 0.2 Minimize resource consumption
Safety Margin 0.2 - 0.3 Maintain safety buffer
User Satisfaction 0.1 - 0.2 Inferred user preference alignment

3.3 Salience Engine

The salience engine computes and updates memory importance scores:

Salience Computation

S(m) = w₁·EMV_impact(m) + w₂·risk_factor(m) + w₃·recency(m) + w₄·frequency(m)

Where:
  EMV_impact(m) = normalized absolute impact on expected value
  risk_factor(m) = amplification for risk-related memories
  recency(m) = exponential decay based on time since creation
  frequency(m) = log(access_count + 1) / log(max_access + 1)

Risk Factor 0.25 Safety-critical memories amplified
Recency 0.25 Recent experiences more relevant

3.5 Decay and Compression

Memory lifecycle management through decay and compression:

Decay Functions

Exponential:  S(t) = S₀ · e^(-λt)         [Default for episodic]
Power Law:    S(t) = S₀ · (1 + t)^(-α)    [For semantic memories]
Linear:       S(t) = max(S_min, S₀ - kt)  [For time-bounded tasks]
Step:         S(t) = S₀ if t < T else 0   [For deadline-based events]

Compression Strategy

When memory capacity approaches limits, compression is triggered:

  1. Identify Candidates — Memories with salience below compression threshold
  2. Group by Type — Cluster similar low-salience memories
  3. Generate Summary — Create compressed representation preserving key insights
  4. Supersede Originals — Replace individual memories with summary
  5. Update References — Maintain provenance chain to originals

def process_observation(self, raw_obs: RawObservation) -> ProcessedObservation:
    # 1. Validate source

        raise UntrustedSourceError(raw_obs.source)

    # 2. Normalize values
    normalized = self.normalize(raw_obs.values)

    imputed = self.impute_missing(normalized)

        values=weighted,
        timestamp=datetime.now(),
        source=raw_obs.source,
        confidence=self.compute_confidence(weighted)
    )

Update Methods

Bayesian Belief Updater
P(θ|D) ∝ P(D|θ) · P(θ)

posterior = likelihood × prior / evidence

Used for: Discrete states, categorical variables, hypothesis testing

x̂ₖ = x̂ₖ₋₁ + Kₖ(zₖ - Hx̂ₖ₋₁)

V. EMV Utility Framework

The EMV Utility Framework provides the economic reasoning capability for the agent, enabling value-based decision-making with appropriate risk management.

5.1 EMV Calculator

emv_utility/emv_calculator.py

Core Calculation

EMV(a) = Σᵢ P(oᵢ | a) × V(oᵢ)

Where:
  a = action under consideration
  oᵢ = possible outcome i
  P(oᵢ | a) = probability of outcome given action
  V(oᵢ) = monetary value of outcome

Outcome Enumeration

The OutcomeEnumerator generates the probability-weighted outcome set:

class OutcomeEnumerator:
    def enumerate(
        self,
        state: State,
        action: Action
    ) -> List[WeightedOutcome]:

        # Generate base outcomes from transition model
        base_outcomes = self.transition_model.predict_outcomes(state, action)

        # Expand with uncertainty
        expanded = self.expand_uncertainty(base_outcomes)

        # Normalize probabilities
        total_prob = sum(o.probability for o in expanded)
        for outcome in expanded:
            outcome.probability /= total_prob

        return expanded

Action Comparison

def compare_actions(
    self,
    state: State,
    actions: List[Action]
) -> ActionRanking:

    emv_scores = {}
    for action in actions:
        emv_scores[action] = self.calculate_emv(state, action)

    ranked = sorted(
        emv_scores.items(),
        key=lambda x: x[1],
        reverse=True
    )

    return ActionRanking(
        best_action=ranked[0][0],
        scores=emv_scores,
        margin=ranked[0][1] - ranked[1][1] if len(ranked) > 1 else float('inf')
    )

class AdaptiveUtilityFunction:
    def compute(self, value: float, context: Context) -> float:
        # Increase risk aversion when:
        # - Stakes are high
        # - Resources are low
        # - Time pressure is high

        risk_aversion = self.base_risk_aversion

        if context.stakes > self.high_stakes_threshold:
            risk_aversion *= 1.5

        if context.resource_availability < 0.3:
            risk_aversion *= 1.3

        return self.power_utility(value, risk_aversion)

5.3 Risk Model

emv_utility/risk_model.py

Risk Measures

Value at Risk (VaR)

The maximum loss not exceeded with a given confidence level.

VaR_α = -inf{x : P(X ≤ x) ≥ α}

Example: VaR₉₅ = $10,000 means 95% confident loss won't exceed $10,000

Conditional Value at Risk (CVaR)

Expected loss in the worst α% of cases.

CVaR_α = E[X | X ≤ VaR_α]

More conservative than VaR—captures tail risk.

Risk Categories

Category Examples Mitigation Focus
Operational System failure, timeout, resource exhaustion Redundancy, monitoring
Financial Cost overrun, opportunity cost, market volatility Budgets, hedging
Compliance Policy violation, regulatory breach Constraints, auditing
Reputational User dissatisfaction, trust erosion Quality controls, escalation
Strategic Goal misalignment, suboptimal outcomes Alignment checks, oversight

6.1 Learning Update Engine

learning/learning_update_engine.py

Learning Algorithms

Temporal Difference Learning
Q(s,a) ← Q(s,a) + α[r + γ·max_a' Q(s',a') - Q(s,a)]

Where:
  α = learning rate
  γ = discount factor
  r = immediate reward
  s' = next state
Policy Gradient
∇J(θ) = E[∇log π_θ(a|s) · A(s,a)]

Where:
  θ = policy parameters
  π_θ = parameterized policy
  A(s,a) = advantage function

Salience-Based Learning

Learning rates are modulated by experience salience:

α_effective = α_base × salience(experience)

# High-salience experiences (critical outcomes, errors)
# have stronger learning effect

Experience Replay

class ExperienceReplay:
    def __init__(self, capacity: int = 10000):
        self.buffer = deque(maxlen=capacity)

    def sample_batch(self, batch_size: int) -> List[Experience]:
        # Priority sampling based on TD error
        priorities = [self.compute_priority(exp) for exp in self.buffer]
        probs = np.array(priorities) / sum(priorities)

        indices = np.random.choice(
            len(self.buffer),
            size=batch_size,
            p=probs
        )

        return [self.buffer[i] for i in indices]

Enforcement Flow

def check_constraints(self, state: State, action: Action) -> EnforcementResult:
    violations = []

    # Check constraints in priority order
    for constraint in sorted(self.constraints, key=lambda c: c.priority, reverse=True):
        if constraint.is_violated(state, action):
            violations.append(self.create_violation(constraint, state, action))

            if constraint.type == ConstraintType.HARD:
                # Hard constraint - block immediately
                return EnforcementResult(
                    allowed=False,
                    violations=violations,
                    action=EnforcementAction.BLOCK
                )

    # All hard constraints passed
    return EnforcementResult(
        allowed=True,
        violations=violations,
        warnings=[v.message for v in violations]
    )

Appendix C: Security Considerations

C.1 Threat Model

Threat Mitigation Detection
Prompt Injection Input sanitization, structured prompts Pattern matching, anomaly detection
Memory Tampering Integrity checks, signed memories Hash verification, provenance audit
Privilege Escalation Authority level enforcement Action logging, boundary alerts
Data Exfiltration Output filtering, classification Content scanning, rate limiting
Denial of Service Resource limits, rate limiting Usage monitoring, threshold alerts

C.2 Security Hardening

  • Principle of Least Privilege — Grant minimum necessary authority
  • Defense in Depth — Multiple layers of constraint enforcement
  • Fail Secure — Default to safe state on errors
  • Complete Mediation — Check constraints on every action
  • Separation of Duties — Different agents for different functions

C.3 Incident Response

  1. Detection — Automated alerts on constraint violations
  2. Containment — Automatic escalation and potential shutdown
  3. Investigation — Decision trace analysis
  4. Remediation — Constraint updates, policy adjustments
  5. Recovery — Controlled restart with enhanced monitoring
  6. Lessons Learned — Update failure analysis module
